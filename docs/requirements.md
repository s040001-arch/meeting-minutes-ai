# Requirements / MVP Specification

## 0. Overview
議事録作成に必要な「音声入力 → 文字起こし → 自動補正（ルールベース＋AI）→ 不明点の質問ループ → 構造化出力（Google Docs）→ 学習・再利用」までを、MVPとして順番に実装するための要件定義。

## 1. Identifiers & Status (共通定義)
### 1.1 Identifiers
- `job_id`: 1ジョブを一意に識別するID（同時処理は1件のみ想定）
- `chunk_id`: 音声チャンク単位の識別子
- `question_id`: 質問ループで内部的に管理するID

### 1.2 Status
- 音声入力処理：`invalid_filename`（命名規則不一致時）、重複時はスキップ
- 文字起こし：`success` / `failed`
- 文字起こし失敗時：`error` に理由（例：`timeout`）を保持

## 2. ① 入力・データ取得（Audio Intake / Drive Polling）
音声ファイルを取得し、処理可能な状態にする。

### 2.1 機能要件
- 録音元：`Pixel 9` 標準レコーダー または `Notta` アプリ
- 入力：音声ファイルのみ（テキスト入力は扱わない）
- ユーザーは指定の `Google Driveフォルダ` に音声を手動アップロードする
- システムは `フォルダを定期巡回（例：5分間隔）` する
- 新規ファイル検出時：ユーザー確認なしで即時処理開始
- 音声の最大長：最大 `90分` を想定
- サーバー側で `30〜60秒` 単位に分割する前提
- 同時処理：`1ジョブのみ`（MVP）
- 各音声は `job_id` で一意管理

### 2.2 入出力
- 入力
  - 音声ファイル（`m4a / mp3 / wav`）
  - Google Drive上の指定フォルダ
- 出力
  - 保存された音声データ
  - `job_id`
  - メタ情報（後工程用）

### 2.3 ファイル命名規則
- フォーマット：`yyyy_mmdd_顧客名_会議名`
- 例：`2026_0323_ABC商事_定例会議.m4a`

### 2.4 命名規則の処理
- ファイル名から抽出する
  - 日付
  - 顧客名
  - 会議名
- 命名規則に一致しない場合
  - 処理対象外とする
  - ステータス：`invalid_filename`

### 2.5 重複処理ルール
- 同一ファイル名の音声が既に存在する場合
  - 処理をスキップする

### 2.6 音声形式処理
- 対応形式：`m4a / mp3 / wav`
- サーバー内部では：すべて `wav` 形式に変換して処理する

### 2.7 制約
- 最大 `90分` 対応のため、分割処理前提
- 同時実行は `1件` に制限
- `Drive API` の制限を考慮し、過剰ポーリングしない

## 3. ② 文字起こし（一次変換 / Whisper）
音声データをテキストに変換し、後続の補正処理に利用可能な状態にする。

### 3.1 機能要件（全体）
- 事前に `30秒単位` でチャンク分割される
- 各チャンクごとに Whisper系エンジンで文字起こしを実行
- 文字起こし結果は `チャンク単位` で保存する
- 各チャンクには以下の情報を保持する
  - `chunk_id`
  - `job_id`
  - 開始時間 / 終了時間
  - テキスト
  - ステータス（`success / failed`）
  - エラー情報（失敗時）
- チャンク単位で処理結果をログ出力する

### 3.2 入出力
- 入力
  - 分割済み音声チャンク（`wav`）
- 出力
  - チャンク単位の文字起こしテキスト
  - チャンク単位の処理ログ

### 3.3 テストモード（MVP）
- `Level 1`: 最初の1チャンクのみ処理
- `Level 2`: 最初の2チャンクのみ処理
- `Level 3`: 全チャンク処理（本番モード）

### 3.4 処理制御
- チャンクは順次処理（MVPでは `直列`）
- 将来的に並列処理可能な設計にする（現時点では実装しない）

### 3.5 ログ要件（例：JSON形状）
- `success` の例
  - `{ "chunk_id": "chunk_001", "status": "success", "text": "...", "duration": 30 }`
- `failed` の例
  - `{ "chunk_id": "chunk_002", "status": "failed", "error": "timeout" }`

### 3.6 制約
- 最大 `90分` 対応のためチャンク分割前提
- `1チャンク単位` で再実行可能であること
- 全体失敗ではなく `部分失敗` として扱う

## 4. ③ 自動補正・意味理解（Correction Pipeline）
文字起こし結果の誤りを修正し、文意が通る自然なテキストへ変換する。

### 4.1 処理全体フロー
- チャンク単位のテキストを `時系列順` に結合
- 全文に対して
  - 第1段階補正（機械補正）
  - 第2段階補正（AI補正）
- 不明箇所を抽出
- 必要に応じてユーザーに質問
- 回答を反映し再補正
- 必要なら再度質問ループ

### 4.2 第1段階：機械補正（ルールベース）
軽量・高速で再現性がある修正。
- 表記統一
  - 全角 / 半角 / 記号
- 数字正規化
  - 例：`三` → `3`
- 辞書ベース置換
  - 顧客名
  - 製品名
  - 人名
  - 社内用語
- 明らかな誤字修正

### 4.3 第2段階：AI補正（意味理解）
文脈に基づく誤変換修正。
- 内容
  - 文脈に基づく誤変換修正
  - 同音異義語の補正
  - 不自然な文章の修正
  - 文の再構成
  - 会話的表現 → 議事録向け表現への変換
- 補正単位
  - `全文` をまとめて1回（チャンク単位では実施しない）
- 不明箇所の抽出（検出するもの）
  - 文意が通らない箇所
  - 固有名詞が曖昧
  - 数字・数量が不明確
  - 主語・対象が不明
  - 文脈的に矛盾している箇所
- 補正ループ
  - AI補正後に不明点がある場合：質問生成へ進む
  - ユーザー回答後：再度全文補正を実行
  - 必要に応じて繰り返す
- 制約
  - 補正は `1回で完結させない`（ループ前提）
  - 第1段階 → 第2段階の順序を必ず守る

## 5. ④A：事前情報・補足情報投入（Context Notes）
議事録作成の精度を高めるため、ユーザーが任意で背景情報や補足情報を入力できるようにする。

### 5.1 機能要件
- ユーザーは任意で補足情報を入力できる
- 補足情報がなくても議事録作成を開始できる
- 補足情報は開始前に入力できる
- 補足情報は処理途中にも追加できる
- 補足情報は自由文で入力する
- 補足情報は `context_note` として保存する
- 補足情報は補正処理時に参照する

### 5.2 入力例
- 会議の背景
- 顧客名の正式表記
- 案件の前提
- よく出る略語
- 今回重視したい観点
- 過去経緯

### 5.3 制約
- 必須入力ではない
- 毎回入力を強制しない
- 補足情報がなくても処理は継続できる

## 6. ④B：質問ループ（Unknown Clarification Loop）
AIだけでは確定できない不明点について、ユーザーに確認し、議事録精度を高める。

### 6.1 ループ開始条件
- 質問は初回の全文補正完了後に開始する
- AIは補正後テキストから不明箇所を抽出する

### 6.2 優先度付けと選択ルール
- 各不明箇所に重要度スコアを付与する
- 未解消の不明箇所のうち重要度が最も高い `1件` を質問する

### 6.3 入出力（自由文）
- 質問は自由文で行う
- ユーザー回答も自由文で受け取る

### 6.4 管理ルール
- 質問は1回に1件のみ送信する
- 未回答の質問は常に1件のみ
- 各質問は内部的に `question_id` で管理する
- 回答受領後、回答内容を反映して再度全文補正を実行する
- 再補正後に不明箇所を再抽出し、再度スコアリングする
- このループを最大 `3回` まで繰り返す
- `3回` を超えても不明点が残る場合は、その時点の最善結果を出力する
- MVPでは質問送信チャネルを `LINE` 固定とする

### 6.5 質問文に含める内容
- 不明箇所の引用
- 前後の文脈
- なぜ不明なのか
- 何を教えてほしいのか

### 6.6 重要度スコアの観点
- 全文の意味に与える影響
- 顧客名、案件名、製品名、人名、金額、日付などの重要情報か
- 回答により複数箇所をまとめて修正できるか
- そのままだと議事録品質を大きく損なうか
- 補足情報や過去知識でも解消できないか

### 6.7 制約
- 複数質問を同時送信しない
- 選択式ではなく自由文質問
- MVPではLINE以外のチャネルは対象外

## 7. ⑤ 出力・学習・再利用（Google Docs Output）
補正済みテキストを議事録として構造化し、Google Docsに出力する。

### 7.1 出力先
- Google Docs

### 7.2 Google Docsファイル名
- 音声ファイルと同じ命名規則を使用
- フォーマット：`yyyy_mmdd_顧客名_会議名`

### 7.3 出力形式（固定構造）
- タイトル（書式＝タイトル）
- 参加者（見出し1）
- 会議概要（見出し1）
- 決まったこと（見出し1）
- 残論点（見出し1）
- Next Action（見出し1）
- 発言録（見出し1）

### 7.4 記載順序（最重要）
- 必ず以下の順序で生成する
  - 発言録
  - その他項目（参加者、概要、決定事項など）

### 7.5 発言録生成ルール（超重要）
- 元の発言内容を極力そのまま記載
- 許可する編集
  - フィラー削除（例：`えーっと`、`あのー`）
  - 明らかな誤字修正
  - 文の区切り整理
- 禁止する編集
  - 意味の補完
  - 推測による書き換え
  - 内容の要約

### 7.6 他セクション生成ルール
- 発言録とファイル名情報を元に生成する
- 推察は最小限にする
- 不明確な情報は無理に補完しない
- 発言録と矛盾する内容を記載しない

### 7.7 制約（特に重要）
- 発言録はAIが想像で内容を補完することを禁止
- 不確実な情報は断定しない

## 8. ⑥ 学習・再利用（Knowledge Accumulation & Reuse）
過去の補足情報や質問回答などを知識として蓄積し、発言録の補正時に文脈解釈や固有名詞の推定に活用する。

### 8.1 基本方針
- 学習の起点は発言録の補正
- 過去情報は補正の参考材料として利用
- 厳密な分類ではなく、文脈補正のための知識として保持

### 8.2 保存対象
- ユーザーが入力した補足情報（`context_note`）
- LINEでの質問に対するユーザー回答
- ファイル名から取得した顧客名・会議名
- 意味解釈を伴わない表記統一情報
- ユーザー確認済みの固有名詞・用語

### 8.3 自動保存しない情報
- AIが文脈から推定した情報
- AIが生成した要約（会議概要、決定事項など）
- 意味解釈を伴う補正結果
- ユーザー未確認の固有名詞

### 8.4 保存形式（Googleスプレッドシート）
- 学習データはGoogleスプレッドシートで管理する
- 各データは1行単位で記録する
- スプレッドシートのイメージ（列）
  - 種類（補足情報 / 回答 / 辞書）
  - 内容
  - 元の文脈（任意）
  - 登録日時

### 8.5 再利用方法
- 発言録の補正時に、現在の文章や会議名・顧客名に関連しそうな過去情報を参照する
- 参照した情報をもとに以下を行う
  - 固有名詞補正
  - 用語解釈
  - 文脈補完
  - 表記統一
- 過去情報は参考として扱い、無条件に採用しない
- 現在の発言録の内容と矛盾する場合は現在を優先する

### 8.6 管理方法 / 制約
- ユーザーはスプレッドシートを直接編集できる
- 不要な情報は削除できる
- 誤った情報は修正できる
- 制約
  - AIの推測情報は学習しない
  - ユーザー情報を優先する
  - 過去情報と現在の文脈が矛盾する場合は現在を優先する
  - 学習データがなくても処理は可能

## 9. MVPタスク一覧（実装順）
MVPは「入力 → 音声処理 → 文字起こし → 補正 → 質問ループ → 出力 → 学習」の順で作る。

### Phase 1: 入力（最初の壁）
1. Task 1-1：Google Drive接続
   - 指定フォルダにアクセスできる
   - ファイル一覧取得
   - 成功条件：ファイル名が一覧で取れる
2. Task 1-2：新規ファイル検出
   - 前回との差分を検出
   - 新規ファイルだけ取得
   - 成功条件：新しく追加したファイルだけ検出できる
3. Task 1-3：音声ダウンロード
   - Driveからローカルに保存
   - 成功条件：`mp3/m4a/wav` が保存される

### Phase 2: 音声処理（ここで詰まりやすい）
1. Task 2-1：ffmpegでwav変換
   - 全ファイルをwavに変換
   - 成功条件：どの形式でもwavになる
2. Task 2-2：30秒チャンク分割
   - 音声を分割
   - 成功条件：`chunk_001.wav`, `chunk_002.wav` ができる
3. Task 2-3：テストモード
   - 1チャンクだけ処理
   - 2チャンクだけ処理
   - 成功条件：チャンク数制御できる

### Phase 3: 文字起こし
1. Task 3-1：Whisper接続
   - 1チャンクだけ文字起こし
   - 成功条件：テキストが返る
2. Task 3-2：チャンク単位保存
   - JSON or DBに保存
   - 成功条件：チャンクごとにテキスト保存
3. Task 3-3：ログ出力
   - success / failed 記録
   - 成功条件：どこで失敗したか分かる

### Phase 4: 補正（ここがコア）
1. Task 4-1：チャンク結合
   - 全文作成
   - 成功条件：1つのテキストになる
2. Task 4-2：機械補正
   - 表記統一
   - 辞書置換
   - 成功条件：軽い補正ができる
3. Task 4-3：AI補正
   - 文意修正
   - 成功条件：読める文章になる
4. Task 4-4：不明箇所抽出
   - AIに「分からない所」を出させる
   - 成功条件：不明点リストが出る

### Phase 5: 質問ループ
1. Task 5-1：質問生成
   - 1件だけ生成
   - 成功条件：質問文が出る
2. Task 5-2：LINE送信
   - Botで送る
   - 成功条件：LINEに届く
3. Task 5-3：回答受信
   - webhookで受け取る
   - 成功条件：回答が取得できる
4. Task 5-4：再補正
   - 回答を反映
   - 成功条件：テキストが更新される

### Phase 6: 出力
1. Task 6-1：発言録生成
   - 成功条件：ほぼそのまま出る
2. Task 6-2：他セクション生成
   - 成功条件：構造ができる
3. Task 6-3：Google Docs出力
   - 成功条件：Docsに書き込まれる

### Phase 7: 学習
1. Task 7-1：スプレッドシート接続
   - 成功条件：読み書きできる
2. Task 7-2：保存処理
   - 成功条件：回答が保存される
3. Task 7-3：参照処理
   - 成功条件：補正時に読み込める

## 10. 最重要ルール（この順守で成功）
- `① 一気に作らない`: 1タスクずつ
- `② 毎回「成功条件」を確認`: 動いたら次へ
- `③ テストモードを使う`: 待たない
- `④ ログを必ず見る`: ブラックボックス禁止
